# Спецификация: Устойчивый скраппинг HLTV (анти-бот защита)

## Контекст

Текущий скраппер (`Scrapers::Hltv::Teams`) уже использует Selenium + headless Chrome с базовыми настройками (`--disable-blink-features=AutomationControlled`, кастомный User-Agent). Однако каждый вызов создаёт и уничтожает драйвер отдельно, нет управления задержками и нет обработки блокировок. Для начальной загрузки (~20–30k запросов) это неприемлемо.

Стек: Rails 8, Selenium WebDriver + Chrome, Nokogiri. Новые гемы — только по явному согласованию.

---

## Функциональные требования

### FR-1. Случайные задержки между запросами

Между каждым HTTP-запросом к HLTV должна быть задержка с рандомизацией, имитирующая человеческое поведение:

- Базовый диапазон: **3–8 секунд** между запросами одного скраппера
- Периодические «перерывы»: каждые **20–30 запросов** — пауза **30–90 секунд**
- Диапазоны настраиваются через конфигурацию (константы в классе, не хардкод)

### FR-2. Повторное использование браузерной сессии

Вместо создания нового экземпляра `Selenium::WebDriver` на каждый запрос:

- В рамках одного «сеанса скраппинга» (одного вызова job'а) используется единственный драйвер
- Драйвер закрывается только по завершении сеанса или при критической ошибке
- Сессия сохраняет cookies между запросами (как в реальном браузере)

### FR-3. Обнаружение блокировки и backoff

Скраппер должен уметь распознать, что HLTV заблокировал запрос:

- Условия обнаружения: HTTP 403/429, наличие в DOM признаков CAPTCHA или challenge-страницы (например, Cloudflare), отсутствие ожидаемого контента после таймаута
- При обнаружении блокировки: **не бросать исключение сразу**, а выдержать паузу и повторить попытку
- Стратегия повтора: до **3 попыток** с экспоненциальным backoff (30s → 90s → 300s)
- После 3 неудачных попыток — бросить `Scrapers::Hltv::Error` с описанием причины

### FR-4. Пакетная обработка для начальной загрузки

Для сценария ~20–30k запросов:

- Скраппинг разбивается на **пакеты** (batch), размер настраивается (по умолчанию: 50 запросов)
- Между пакетами — пауза **5–15 минут**
- Прогресс пакетной загрузки персистируется в БД (чтобы при перезапуске не начинать с нуля)

### FR-5. Централизованный rate limiter

Все классы `Scrapers::Hltv::*` должны соблюдать единый лимит параллельных запросов к HLTV:

- Не более **1 запроса одновременно** (последовательная очередь)
- Реализация через **PostgreSQL Advisory Lock** — это обеспечивает взаимное исключение между несколькими Solid Queue-воркерами (отдельные OS-процессы), что mutex в рамках одного Ruby-процесса не гарантирует

## Нефункциональные требования

- **NFR-1. Тестируемость.** Задержки и backoff-логика должны быть легко заглушаемы в тестах (через dependency injection или конфигурацию).
- **NFR-2. Наблюдаемость.** Каждая задержка, повтор и блокировка логируются через стандартный `Rails.logger` с уровнем `info`/`warn`.
- **NFR-3. Без новых гемов** без явного согласования. Реализация на стандартном Ruby + Selenium + Rails.

## Критерии приёмки

| # | Критерий |
|---|----------|
| AC-1 | Начальная загрузка 20–30k запросов завершается без блокировки (ручная проверка) |
| AC-2 | За первые 2 недели production-эксплуатации в логах нет событий уровня `warn` с признаком блокировки (нет 403/429, нет CAPTCHA-детекции, нет исчерпания retry) |
| AC-3 | При симуляции 403/CAPTCHA скраппер делает 3 повтора с паузами и только потом бросает ошибку |
| AC-4 | В тестах задержки не приводят к замедлению (`sleep` заглушён) |
| AC-5 | Прогресс пакетной загрузки сохраняется: после перезапуска job продолжает с последнего успешного пакета |
| AC-6 | Два параллельных скраппера не отправляют запросы к HLTV одновременно (проверяется интеграционным тестом с двумя тредами, конкурирующими за advisory lock) |
| AC-7 | Пост-релизный мониторинг: отсутствие блокировок через 3 месяца после запуска регулярного сбора (фиксируется в задаче ревью) |

## Архитектурное решение

### Новые компоненты

```
lib/scrapers/hltv/
  browser_session.rb     # управление жизненным циклом Selenium-драйвера
  rate_limiter.rb        # глобальный advisory lock (PostgreSQL)
  request_throttle.rb    # логика задержек и backoff
  base_scraper.rb        # базовый класс для всех Hltv-скрапперов

app/models/
  scraper_checkpoint.rb  # персистирование прогресса пакетной загрузки (новая таблица)

app/jobs/
  hltv_bulk_import_job.rb  # управляет пакетной загрузкой
```

### Изменения в существующем коде

- `Scrapers::Hltv::Teams` наследуется от `BaseScraper` (вместо прямого использования Selenium)
- `fetch_html` делегирует браузерные операции в `BrowserSession`
- Метод `call` оборачивается в `RequestThrottle` и `RateLimiter`

### Таблица `scraper_checkpoints`

```
id, scraper_name (string), last_completed_batch (integer),
total_batches (integer), status (string), created_at, updated_at
```

## Out of scope

- Ротация прокси (потребует нового гема или внешнего сервиса — отдельное решение)
- Полноценная эмуляция движений мыши / скролла (усложняет код непропорционально пользе)
- Автоматическое решение CAPTCHA
- Мониторинг блокировок в продакшене (отдельная задача по observability)
