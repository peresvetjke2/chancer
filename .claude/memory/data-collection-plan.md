# План реализации: Сбор данных

## Фаза 0 — Инициализация проекта

- [x] Создать Rails 8 приложение (`rails new chancer --database=postgresql`)
  - **Критерий:** `bin/rails s` запускается без ошибок, открывается стартовая страница
- [x] Настроить тестовое окружение (RSpec, FactoryBot)
  - **Критерий:** `bundle exec rspec` проходит на пустом проекте без ошибок
- [x] Подключить базу данных PostgreSQL
  - **Критерий:** `bin/rails db:create` выполняется успешно, базы `chancer_development` и `chancer_test` существуют

## Фаза 1 — Модели данных

- [ ] Модель `Team` (название, регион, рейтинг)
  - **Критерий:** миграция применена, модель проходит валидации, есть фабрика FactoryBot
- [ ] Модель `Player` (имя, роль, ссылка на команду)
  - **Критерий:** миграция применена, ассоциация `belongs_to :team` работает, есть фабрика
- [ ] Модель `Match` (команды, дата, турнир, результат, счёт)
  - **Критерий:** миграция применена, ассоциации на две команды работают, есть фабрика
- [ ] Модель `MapResult` (карта, счёт, ссылка на матч)
  - **Критерий:** миграция применена, `belongs_to :match` работает, есть фабрика
- [ ] Модель `PlayerStats` (K/D, рейтинг игрока в матче)
  - **Критерий:** миграция применена, ассоциации на `player` и `match` работают, есть фабрика
- [ ] Модель `NewsItem` (источник, дата, текст)
  - **Критерий:** миграция применена, есть фабрика, поле `source` обязательно

## Фаза 2 — Парсеры / источники данных

- [ ] Базовый класс / интерфейс скрапера (`app/services/scrapers/base_scraper.rb`)
  - **Критерий:** все скраперы наследуются от него, есть единый метод `call`, покрыт unit-тестами
- [ ] Парсер HLTV — матчи и рейтинги
  - **Критерий:** при вызове сохраняет ≥1 `Match` и ≥1 `Team` в базу, покрыт тестами с VCR/WebMock
- [ ] Парсер HLTV — статистика игроков
  - **Критерий:** при вызове сохраняет `PlayerStats` для матча, покрыт тестами с VCR/WebMock
- [ ] Парсер Liquipedia — составы и трансферы
  - **Критерий:** при вызове обновляет `Player` и их принадлежность к `Team`, покрыт тестами
- [ ] Парсер новостей (RSS)
  - **Критерий:** при вызове сохраняет `NewsItem`, дубликаты не создаются (idempotent), покрыт тестами

## Фаза 3 — Фоновая очередь

- [ ] Подключить Solid Queue (входит в Rails 8)
  - **Критерий:** `bin/jobs` запускается, очередь обрабатывает задачи
- [ ] `FetchMatchHistoryJob`
  - **Критерий:** запускается вручную через `perform_now`, создаёт записи в `Match`
- [ ] `FetchRosterJob`
  - **Критерий:** запускается вручную, обновляет `Player` и `Team`
- [ ] `FetchNewsJob`
  - **Критерий:** запускается вручную, создаёт `NewsItem`
- [ ] Расписание через `config/recurring.yml`
  - **Критерий:** после перезапуска сервера задачи выполняются автоматически по расписанию

---

## Фаза 4 — Надёжность и антибан

- [ ] Rate limiting и задержки между запросами в скраперах
  - **Критерий:** при запуске нескольких задач подряд интервал между HTTP-запросами к одному хосту ≥ N мс (настраивается)
- [ ] Retry-логика при сетевых ошибках
  - **Критерий:** при временной ошибке (5xx, таймаут) задача повторяется автоматически, есть лог повтора
- [ ] Логирование ошибок парсеров
  - **Критерий:** при падении парсера ошибка записывается в лог с достаточным контекстом (источник, время, исключение)

## Фаза 5 — API для просмотра собранных данных

- [ ] Эндпоинт `GET /api/teams`
  - **Критерий:** возвращает список команд в JSON, работает пагинация
- [ ] Эндпоинт `GET /api/matches`
  - **Критерий:** возвращает список матчей с фильтрацией по команде и дате
- [ ] Эндпоинт `GET /api/news`
  - **Критерий:** возвращает список новостей, отсортированных по дате
