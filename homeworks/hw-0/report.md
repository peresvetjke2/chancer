# Отчёт: разработка через AI-агента

## Что делали

Запускали агента на составление верхнеуровневого плана реализации MVP для проекта Chancer (AI-помощник для прогнозирования ставок на киберспорт). Итоговый артефакт — `.claude/memory/plans/general.md`: 5 фаз, 22 задачи, критерий приёмки на каждую.

## Как агент работает с кодом

- **Хорошо следует структуре промпта.** Когда явно задан Context/Constraints/Output — агент точно воспроизводит требуемый формат и не фантазирует.
- **Читает файлы проекта перед ответом.** После указания на PROJECT.md агент самостоятельно открыл файл и выстроил план на основе реальной функциональности, а не придуманной.
- **Атомизирует задачи.** Разбивает крупные цели на конкретные шаги с измеримым результатом — удобно как чеклист для дальнейшей работы.
- **Сохраняет артефакты туда, куда сказано.** Создал папку `.claude/memory/` и корректно записал план без лишних вопросов.

## В чём не доверяю агенту

- **Миграции базы данных.** Агент может добавить или изменить поля, не соблюдая соглашение "не трогать существующие миграции". Всегда проверяю `db/migrate/` после его работы.
- **Выбор гемов.** Склонен предлагать новые зависимости там, где можно обойтись стандартными средствами Rails. Без явного ограничения "No new gems" в CLAUDE.md — подтянет что-нибудь лишнее.
- **Переусложнение архитектуры.** На этапе планирования предложил сервис-объекты и слой абстракций раньше, чем это нужно MVP. Потребовалось явно ограничить: "Standard Rails MVC, no service objects yet".
- **Критерии приёмки без уточнения.** В первых итерациях агент писал план без критериев — пока не добавил Constraints в промпт. Сам он это не инициирует.

## Вывод

Агент эффективен как генератор структуры и шаблонного кода, но требует точного промпта с явными ограничениями. Без них — расширяет scope и усложняет решение.
